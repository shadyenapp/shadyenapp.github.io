Introduction

        The Baldwin County Health Department needs a MySQL service and workbench for Frank, Adin, and Pamela to ultimately provide insightful reports to local clinics. As the bastion of public health services in Alabama, they will be using Ubuntu 22.04, an open-source Linux distribution to keep costs low and ensure that the reports generated by the Covid19 data will be available to access for each local clinic no matter the operating system. This system also needs to remain secure from potential attack, and as such it is important that not only the MySQL service is secure, but the operating system is as well. The local users and MySQL users on this system will abide by the principle of least privilege and only have access to the objects they need access to.

Sanitization

Before any data can be loaded into the MySQL instance, it first must be sanitized and checked for SQL statements, so nothing gets executed as data is loaded in. If there were SQL statements in the data that were unnoticed, that could leave security holes within the database and surrounding environment which could potentially give attackers full access to the system. To begin data sanitization, a shell script is made to iterate through a list of SQL commands and search a given file for any results of those commands, shown in figure 24. Since the data that is expected to be loaded into the database is health-related data, there should not be anything having to do with these keywords specifically—if the data had included these statements as valid data, a new method would need to be made to sanitize the data. From running the script, shown in figure 25, the SQL statements identified are removed from the csv file before loading.

Log Analysis

Following Pamela’s queries, an analysis of the general log files needs to be conducted. As shown in figure 31, Pamela’s queries are logged with the time, date, process ID, and user ID. In the MySQL workbench user configuration, the user with user ID 44 is Pamela, so these logged items are completely normal and show that the log settings and variables set earlier work as intended. Additionally, in figure 32, the user ID of 23—the database administrator—was the one who created the table data to load in the MySQL instance. Again, this shows that the log is working as intended and logging important events like CREATE, and queries.

Unlike Microsoft SQL, MySQL does not have a built-in log viewer/auditor, so to view and go through the logs, it must be done manually by whoever has access to it. As shown in figure 33, only the MySQL user has access to write to the log file, with users associated with the “adm” group able to read it. The adm group is the group that MySQL uses to associate with the logging administrator, which Frank has been assigned to. As there is no built-in tool for viewing and automating these logs, they need to be checked on a timely basis. Given the importance of this data and the health department, these logs should be checked every week by the logging administrator. When doing the weekly checks on the logs, Frank will need to be on the lookout for abnormal activity. As there is no feature to automatically go over the logs, there is not a way to easily make a constantly updating baseline, making it harder to identify items that are outside the baseline.

As automatic detection would not be possible in this scenario, Frank and other logging administrators will need to be extra vigilant to logged events that would be considered a high priority. As it takes time and effort to check the logs each week, the effort needs to be put into items with the highest priority and the highest risk to the environment. These kinds of items include CREATE queries, DROP queries, FLUSH queries, and ALTER queries. These queries—if executed successfully—could allow an attacker or insider to not only damage the potentially classified data but destroy the health department’s business continuity. The health department would need lots of time to repair, fix and mitigate the threat, which as a health department can be potentially fatal to the people who won’t get care in that period. Additionally, the logging administrators need to be on the lookout for numerous events in the MySQL error log. If the error log shows that are numerous login failures, that could show that there is someone with physical access to a machine on the network that doesn’t know the password. If the error log also shows numerous errors for query-related statements, this could indicate that there is a problem with the data itself—in which case, all actions on the data need to be stopped so the data can be examined and reconstructed as soon as possible to not hinder business continuity.

If items are found in the general log and error log that are suspicious or show a potential threat to the network environment, the following actions need to be taken. The databases and machines connected to the network need to be isolated to stop the spread of infected or compromised systems. After the databases and machines are isolated, the Baldwin County Health Department must conduct a formal audit on the machines, user access control, database security, and data within the database. Notably, the data within the database will need to be checked, sanitized, and reconstructed. The data was originally found with malicious and missing data within, so it must be assumed that it might be compromised again. This formal audit will ensure that security policies and procedures are followed and not ignored. This is also the starting point for incident response to handle the potential breach of data. During this audit user permissions, local access control, physical security measures, data encryption standards, and potential backup data need to be checked. Ensuring that each account only has the specified access rights will ensure that the principle of least privilege is followed to give the least amount of access to the data. Finally, the accounts themselves need to be checked for any additional accounts on the database and machine. One of the malicious statements initially found in the data was a statement to create a user called “roggerrabbit”, so it must be assumed that this attempt may be tried again for future attacks against the server.
